# 評測結果（2026-07-22 實測）

本輪全部實測數據與判讀。**逐次原始值都列出來，不只中位數**——變異數才是誠實的部分。

總花費約 **$1.6**、約 25 分鐘。

---

## 為什麼分三層

一個常見的誤解是「評測就是測模型能力」。SWE-bench 這類基準測的是 **模型 × harness 的乘積**，
從單一個分數**沒辦法分離兩者的貢獻**——這也是 SWE-agent／Agentless 那些論文在比 scaffold 而非比模型的原因。

本專案的三層各有分工：

| 層 | 測什麼 | 模型的角色 | 成本 |
|---|---|---|---|
| ① 檢索 `hit@k` | **純 harness** | **完全不參與** | $0 |
| ② A/B 消融 | **harness 的邊際貢獻** | 固定，當背景 | 幾分錢 |
| ③ SWE-bench | 模型 × harness | 混在一起，分不開 | $0.24/題 |

**①② 用來評自己的機制，③ 只當外部座標。**

## ① 檢索評測：hit@k / MRR（$0、12 秒、無 LLM）

```bash
go run ./cmd/ingest -root internal/eval/testdata/mem_multihop \
  -eval internal/eval/testdata/mem_multihop/labels.jsonl -k 3 -hops 1
```

語料：12 個互連服務節點（`testdata/mem_multihop/.claw/memory/`），12 題標註，半數多跳。

```
模式                  N    hit@k      MRR
keyword            12     0.50     0.50
embedding           0      n/a      n/a     ← 未配置 embedder，整個模式跳過
keyword+kg         12     1.00     0.69
```

**多跳題的答案節點與查詢字面零重疊**，keyword 在設計上就撈不到（Seeds 只回 score>0），
只有沿 `[[link]]` 擴張的 KG 撈得到。

> ⚠️ **報告時別說「三模式對照」**——`embedding` 那列 N=0，實際只有兩個模式有數字。

**防作弊護欄**（`memeval_test.go`）：

```go
if kw >= 1.0 { // keyword 若滿分表示語料沒真正考多跳
    t.Errorf("keyword 不該滿分（多跳題應漏），got %.2f", kw)
}
```

若有人把語料改簡單到 keyword 也能撈到，測試會失敗。這讓「0.50 vs 1.00」不會因語料退化變成假勝利。

這個結論敢直接講。

## ③ 技能 A/B ×5（`cmd/bench -skill-ab`，模型 claude-haiku-4-5）

情境刻意設計成**模型無法從程式碼推得**的專案慣例（升版本號要同時改 `package.json` 與 `VERSION` 且一致）。

| run | OFF 通過 | OFF 回合 | OFF 成本 | ON 通過 | ON 回合 | ON 成本 | Δ回合 |
|---|---|---|---|---|---|---|---|
| 1 | ✗ | 4 | $0.0143 | ✓ | 5 | $0.0194 | +1 |
| 2 | ✗ | 4 | $0.0147 | ✓ | 4 | $0.0159 | +0 |
| 3 | ✗ | 4 | $0.0142 | ✓ | 4 | $0.0162 | +0 |
| 4 | ✗ | 4 | $0.0142 | ✓ | 5 | $0.0203 | +1 |
| 5 | **✓** | 4 | $0.0153 | **✗** | 5 | $0.0196 | +1 |

| | OFF | ON |
|---|---|---|
| 通過率 | **1/5** | **4/5** |
| 回合中位數 | 4 | 5 |
| 成本中位數 | $0.0143 | $0.0194（**+35%**） |

**判讀：技能改善的是正確性，代價是效率。** 與記憶方向相反——多花 35% 成本與一個回合
（`read_skill` 載入正文本來就要多一輪）換到通過率。

**證據強度：低，只能當觀察。**

- **Fisher 精確檢定雙尾 p = 0.206** ——未達顯著。
- run 5 是**反例**：有技能反而失敗。所以不是穩定的因果，是機率上的改善。
- 要下結論得跑到 n ≥ 20。

> 誠實的說法：「方向明確且與機制預期一致，但 n=5 太小，**不能宣稱統計顯著**。」

## ④ SWE-bench：有成本數據，**沒有分數**

### 已完成：opus 生成 5 題 predictions（$1.21、9 分 45 秒）

| # | instance | 成本 | API 呼叫數 |
|---|---|---|---|
| 1 | astropy-12907 | $0.0720 | 3 |
| 2 | astropy-14182 | $0.9114 | **19** |
| 3 | astropy-14365 | $0.1278 | 5 |
| 4 | astropy-14995 | $0.0620 | 4 |
| 5 | astropy-6938 | $0.0394 | 4 |

每題平均 **$0.243**、**117 秒**（其中約 67 秒是 clone repo）。
→ 推算 30 題約 **$7.3 / 1 小時**，成本與時間都不是障礙。

### ❌ 官方 harness 評測**中途叫停**，故 **opus 沒有 resolved 數字**

**不要在任何場合說「opus 跑了 5 題」然後說不出結果。** 目前唯一有官方判定的是先前的 haiku 那輪：

```
haiku：submitted 5、resolved 1（astropy-6938）、errors 0
```

### 意外觀察：agent 提早收手

`MaxTurns = 40`（`engine.defaultMaxTurns`，且無 `config.json` 覆蓋），但**五題中四題在 3~5 回合就結束**。
不是被切斷，是 agent 自己認為做完了。

推測原因：`-swe-env-setup` 空著 → repo 沒裝依賴 → **跑不了測試 → 沒有回饋訊號可迭代**。
它只能讀 issue、讀幾個檔、寫 patch，然後就無事可做。

第 2 題花了 19 回合，說明**只要有東西可探索它就會探索**——差別在有沒有可驗證的迴圈。

**若此推測成立，提升 SWE-bench 分數的最大槓桿是備妥測試環境，而不是換更強的模型。** 未驗證。

## 方法論：本輪犯過的錯（值得記住）

**① n=1 得出的結論兩次都是錯的**

第一次跑 A/B（各一次）得到「記憶有效、技能無效」。跑到 n=5 才發現技能**其實改善通過率**（1/5→4/5），
只是我盯著回合數與成本看——那是 mem-ab 表格強調的欄位，我順手套用到 skill-ab，**看錯了指標**。

教訓：不同機制改善的維度不同，**先想清楚該看哪個指標**再讀數字。

**② 差點只重測不利的結果**

發現 skill-ab 沒效果後，第一反應是「再跑 5 次確認」——但當時並沒有打算重跑 mem-ab。
那正是「不喜歡的結果就多測幾次，喜歡的就收下」的偏誤。**兩邊都跑到 n=5 才對稱。**

**③ 與既有設計的落差**

`evolve.MinVerifySamples = 10` 已經為調參驗證設了樣本門檻，但 **A/B 這邊沒有同樣的門檻**——
`cmd/bench -mem-ab` 跑一次就給結論。這是真的方法論缺口，未修。

## 已知缺口

- [ ] A/B 沒有樣本數門檻（對比 `MinVerifySamples = 10`），單次跑分即輸出結論
- [ ] `embedding` 檢索模式未配置，三模式對照實際只有兩模式
- [ ] SWE-bench opus 分數未取得（評測叫停）
- [ ] `-swe-env-setup` 空著，SWE-bench 生成階段 agent 無法跑測試（上述推測的根因）
- [ ] skill-ab 需要 n ≥ 20 才能對 1/5→4/5 下結論
